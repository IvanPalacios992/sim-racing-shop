# Guía de Despliegue

## Visión General

El proyecto usa un pipeline de CI/CD completo basado en tres plataformas:

| Plataforma | Rol |
|------------|-----|
| **GitHub Actions** | CI: lint, tests, build y análisis de calidad (SonarCloud) |
| **Vercel** | CD: despliegue automático del frontend (Next.js) |
| **Railway** | CD: despliegue automático del backend (.NET) + PostgreSQL |
| **Upstash** | Redis gestionado en la nube |

### Flujo completo

```
Desarrollador
    │
    │  git push / PR
    ▼
GitHub Actions (CI)
    ├── Backend:  lint → test → build → SonarCloud
    └── Frontend: lint → test → build → SonarCloud
            │
            │  CI verde en main
            ▼
    ┌───────────────────────┐
    │  Vercel               │  ← Detecta push a main/develop
    │  Frontend (Next.js)   │     Deploy automático
    └───────────────────────┘
    ┌───────────────────────┐
    │  Railway              │  ← Detecta push a main/develop
    │  Backend (.NET API)   │     Deploy automático
    │  PostgreSQL 16        │
    └───────────────────────┘
    ┌───────────────────────┐
    │  Upstash              │  ← Redis gestionado (siempre activo)
    │  Redis 7              │
    └───────────────────────┘
```

### Entornos

| Entorno | Branch | Frontend | Backend |
|---------|--------|----------|---------|
| **Producción** | `main` | Vercel Production | Railway Production |
| **Staging** | `develop` | Vercel Preview | Railway Staging |

## 1. Configuración de GitHub

### 1.1 Secrets y Variables del repositorio

Ir a **Settings → Secrets and variables → Actions**.

#### Secrets (valores sensibles, cifrados)

| Secret | Descripción | Cómo obtenerlo |
|--------|-------------|----------------|
| `SONAR_TOKEN` | Token de autenticación de SonarCloud | Ver sección [SonarCloud](#4-configuración-de-sonarcloud) |

#### Variables (valores no sensibles, visibles en logs)

| Variable | Descripción | Valor ejemplo |
|----------|-------------|---------------|
| `SONAR_ORGANIZATION` | Clave de organización en SonarCloud | `mi-organizacion` |
| `NEXT_PUBLIC_API_URL` | URL del backend usada en el build de CI | `https://api.simracingshop.com/api` |

> `NEXT_PUBLIC_API_URL` se usa en el job `frontend-build` del CI. Si no se define, el workflow usa `http://localhost:5000/api` como fallback, lo que puede causar que el build sea válido pero apunte a una URL incorrecta en producción.

### 1.2 Protección de ramas

Ir a **Settings → Branches → Add branch protection rule**.

#### Rama `main`

| Opción | Valor |
|--------|-------|
| Require a pull request before merging | ✅ |
| Required approvals | 1 |
| Dismiss stale pull request approvals | ✅ |
| Require status checks to pass | ✅ |
| Status checks requeridos | `Backend - Build`, `Frontend - Build`, `CI Success` |
| Require branches to be up to date | ✅ |
| Do not allow bypassing the above settings | ✅ |
| Allow squash merging | ✅ (única opción permitida) |
| Allow merge commits | ❌ |
| Allow rebase merging | ❌ |

#### Rama `develop`

| Opción | Valor |
|--------|-------|
| Require a pull request before merging | ✅ |
| Require status checks to pass | ✅ |
| Status checks requeridos | `Backend - Build`, `Frontend - Build` |
| Allow squash merging | ✅ (única opción permitida) |
| Allow merge commits | ❌ |

## 2. Configuración de Vercel (Frontend)

### 2.1 Crear el proyecto

1. Acceder a [vercel.com](https://vercel.com) e iniciar sesión.
2. Hacer clic en **Add New → Project**.
3. Seleccionar el repositorio de GitHub `sim-racing-shop`.
4. En **Root Directory**, escribir `frontend`.
5. Vercel detectará automáticamente Next.js. Verificar que el Framework Preset sea **Next.js**.
6. No modificar los comandos de build por defecto.
7. Hacer clic en **Deploy** (este primer deploy puede fallar sin las variables de entorno — es normal).

### 2.2 Variables de entorno

Ir al proyecto en Vercel → **Settings → Environment Variables**.

Añadir las siguientes variables indicando en qué entornos aplican (Production, Preview, Development):

| Variable | Production | Preview | Valor de ejemplo |
|----------|:----------:|:-------:|-----------------|
| `NEXT_PUBLIC_API_URL` | ✅ | ✅ | `https://api.simracingshop.com/api` (prod) / `https://api-staging.simracingshop.com/api` (preview) |

> Para distinguir entre Production y Preview con valores distintos, crear la variable dos veces: una seleccionando solo **Production** y otra seleccionando solo **Preview**.

### 2.3 Configurar ramas de despliegue

Ir a **Settings → Git**.

| Campo | Valor |
|-------|-------|
| Production Branch | `main` |
| Preview Branches | `develop` y cualquier rama `feature/*` |

Con esta configuración:
- Push a `main` → deploy de **producción** en el dominio principal.
- Push a `develop` → deploy de **preview** con URL única generada por Vercel.

### 2.4 Dominio personalizado (producción)

Ir a **Settings → Domains**.

1. Hacer clic en **Add**.
2. Escribir el dominio (ej. `simracingshop.com`).
3. Vercel muestra los registros DNS a configurar en el proveedor del dominio:

```
Tipo    Nombre    Valor
A       @         76.76.21.21
CNAME   www       cname.vercel-dns.com
```

4. Una vez propagados los DNS (hasta 48h), Vercel provisiona el certificado SSL automáticamente.

### 2.5 Verificar el despliegue

- Abrir la URL de producción y comprobar que la aplicación carga.
- Comprobar en **Deployments** que el último deploy tiene estado **Ready**.
- Verificar que `NEXT_PUBLIC_API_URL` apunta al backend correcto abriendo las DevTools del navegador y comprobando las peticiones de red.

## 3. Configuración de Railway (Backend)

### 3.1 Crear el proyecto

1. Acceder a [railway.app](https://railway.app) e iniciar sesión.
2. Hacer clic en **New Project**.
3. Seleccionar **Deploy from GitHub repo**.
4. Seleccionar el repositorio `sim-racing-shop`.
5. Railway detectará el proyecto. En la configuración del servicio:
   - **Root Directory:** `backend`
   - **Build Command:** `dotnet publish src/SimRacingShop.API --configuration Release --output ./publish`
   - **Start Command:** `dotnet publish/SimRacingShop.API.dll`

### 3.2 Añadir PostgreSQL

1. En el proyecto de Railway, hacer clic en **+ New** → **Database** → **Add PostgreSQL**.
2. Railway provisiona automáticamente una instancia de PostgreSQL 16.
3. En la pestaña **Variables** del servicio de PostgreSQL, copiar el valor de `DATABASE_URL`.

### 3.4 Variables de entorno del backend

Ir al servicio del backend → **Variables**. Añadir todas las variables usando `__` como separador de niveles de configuración de .NET:

#### Conexiones

| Variable | Valor |
|----------|-------|
| `ConnectionStrings__DefaultConnection` | Valor de `DATABASE_URL` del servicio PostgreSQL de Railway |
| `ConnectionStrings__Redis` | Valor de `REDIS_URL` del servicio Redis de Railway |

> Railway permite referencias entre servicios con la sintaxis `${{PostgreSQL.DATABASE_URL}}` en lugar de copiar el valor directamente. Esto actualiza la referencia automáticamente si cambia la URL.

#### JWT

| Variable | Valor |
|----------|-------|
| `JwtSettings__Secret` | String aleatorio de mínimo 32 caracteres (`openssl rand -base64 32`) |
| `JwtSettings__Issuer` | `SimRacingShop` |
| `JwtSettings__Audience` | `SimRacingShop` |
| `JwtSettings__ExpiryMinutes` | `60` |
| `JwtSettings__RefreshTokenExpiryDays` | `7` |

#### Email (Resend)

| Variable | Valor |
|----------|-------|
| `ResendSettings__ApiKey` | API Key de [Resend Dashboard](https://resend.com/api-keys) |
| `ResendSettings__FromEmail` | `noreply@simracingshop.com` (dominio verificado en Resend) |
| `ResendSettings__FromName` | `SimRacing Shop` |
| `ResendSettings__FrontendBaseUrl` | `https://simracingshop.com` |

#### Redis (configuración adicional)

| Variable | Valor |
|----------|-------|
| `Redis__Configuration` | Mismo valor que `ConnectionStrings__Redis` |
| `Redis__InstanceName` | `SimRacing_` |

#### CORS

| Variable | Valor |
|----------|-------|
| `Cors__AllowedOrigins__0` | `https://simracingshop.com` |
| `Cors__AllowedOrigins__1` | `https://www.simracingshop.com` |

#### Admin inicial (solo primer deploy)

| Variable | Valor |
|----------|-------|
| `AdminSeed__Email` | Email del administrador inicial |
| `AdminSeed__Password` | Contraseña segura (mín. 8 chars, mayús + minús + dígito) |
| `AdminSeed__FirstName` | `Admin` |
| `AdminSeed__LastName` | `SimRacing` |

> Tras verificar que el admin se ha creado correctamente, estas variables pueden eliminarse. El seeder no vuelve a actuar si ya existe un admin en el sistema.

#### Stripe (cuando esté integrado)

| Variable | Valor |
|----------|-------|
| `Stripe__SecretKey` | `sk_live_...` en producción / `sk_test_...` en staging |
| `Stripe__WebhookSecret` | `whsec_...` (del dashboard de Stripe) |
| `Stripe__Currency` | `eur` |

### 3.5 Configurar rama de despliegue

Ir al servicio del backend → **Settings → Source**.

| Entorno | Branch |
|---------|--------|
| Producción | `main` |
| Staging | `develop` |

Para tener entornos separados (producción y staging), crear dos servicios de Railway en el mismo proyecto: uno conectado a `main` y otro a `develop`, cada uno con su propia base de datos y variables de entorno.

### 3.6 Dominio personalizado del backend

Ir al servicio backend → **Settings → Networking → Add Domain**.

1. Escribir el subdominio del API (ej. `api.simracingshop.com`).
2. Railway muestra el registro CNAME a configurar:

```
Tipo     Nombre    Valor
CNAME    api       <nombre>.railway.app
```

3. Añadir el registro en el proveedor DNS y esperar la propagación.

### 3.7 Verificar el despliegue

1. En Railway → **Deployments**, comprobar que el último deploy tiene estado **Success**.
2. Abrir `https://api.simracingshop.com/swagger` — debe mostrar la UI de Swagger.
3. Probar `GET /api/auth/me` sin token — debe responder `401 Unauthorized` (el API está funcionando).
4. Comprobar en los **Logs** del servicio que no hay errores de conexión a base de datos ni de seed de admin.

## 4. Configuración de SonarCloud

SonarCloud analiza la calidad del código y la cobertura de tests en cada push y PR.

### 4.1 Crear la organización y los proyectos

1. Acceder a [sonarcloud.io](https://sonarcloud.io) con la cuenta de GitHub.
2. Hacer clic en **+** → **Analyze new project**.
3. Seleccionar la organización de GitHub e importar el repositorio `sim-racing-shop`.
4. SonarCloud crea automáticamente un proyecto. Anotar la **Organization Key** mostrada.

### 4.2 Crear el token de autenticación

1. Ir a **My Account → Security → Generate Tokens**.
2. Crear un token con nombre `github-actions`.
3. Copiar el token generado (solo se muestra una vez).

### 4.3 Añadir el token a GitHub

1. En GitHub → **Settings → Secrets and variables → Actions**.
2. Crear el secret `SONAR_TOKEN` con el valor del token copiado.
3. Crear la variable `SONAR_ORGANIZATION` con la Organization Key de SonarCloud.

### 4.4 Verificar el análisis

Tras el siguiente push, comprobar en SonarCloud → **Projects** que aparecen los dos proyectos:
- `simracing-shop-backend`
- `simracing-shop-frontend`

Cada proyecto mostrará métricas de cobertura, bugs, code smells y vulnerabilidades.

## 5. Pipeline de CI/CD — Funcionamiento

### 5.1 Qué ejecuta el pipeline

El workflow `.github/workflows/ci.yml` se dispara en:
- `push` a `main` o `develop`
- `pull_request` hacia `main` o `develop`

Ignora cambios en `docs/**`, `**.md`, `.gitignore` y `LICENSE`.

```
┌─────────────────────────────────────────────────────────┐
│                    GitHub Actions CI                     │
├───────────────────────┬─────────────────────────────────┤
│      BACKEND          │         FRONTEND                 │
├───────────────────────┼─────────────────────────────────┤
│  backend-lint         │  frontend-lint                   │
│  (dotnet format)      │  (ESLint + TypeScript)           │
│         │             │         │                        │
│  backend-test         │  frontend-test                   │
│  (xUnit + coverage)   │  (Vitest + coverage)             │
│         │             │         │                        │
│  backend-build ───────┤  frontend-build                  │
│  (dotnet publish)  │  │  (next build)                    │
│                    │  │         │                        │
│  backend-sonar     │  │  frontend-sonar                  │
│  (SonarCloud)      │  │  (SonarCloud)                    │
└────────────────────┼──┴─────────────────────────────────┘
                     │
              ci-success
         (requiere todos los anteriores)
```

### 5.2 Versiones usadas en CI

| Tecnología | Versión |
|------------|---------|
| .NET | `10.0.x` |
| Node.js | `22` |
| Java (SonarScanner) | `17` (Temurin) |

### 5.3 Artefactos generados

| Artefacto | Contenido | Retención |
|-----------|-----------|-----------|
| `backend-publish` | Binarios del API (.NET) publicados en Release | Por defecto de GitHub (90 días) |
| `backend-coverage` | Reporte OpenCover de cobertura de tests | Por defecto de GitHub |
| `frontend-build` | Carpeta `.next` compilada | Por defecto de GitHub |
| `frontend-coverage` | Reporte LCOV de cobertura de tests | Por defecto de GitHub |

## 6. Flujo de un Despliegue

### 6.1 Despliegue normal (feature → develop → main)

```bash
# 1. Desarrollar en una feature branch
git checkout develop
git pull origin develop
git checkout -b feature/nueva-funcionalidad

# 2. Desarrollar, commitear y subir
git add .
git commit -m "feat(scope): descripción del cambio"
git push origin feature/nueva-funcionalidad

# 3. Abrir PR en GitHub: feature/* → develop
#    El CI ejecuta lint + test + build + SonarCloud
#    Si el CI está verde y hay aprobación → Squash and merge

# 4. develop se actualiza automáticamente
#    → Vercel despliega preview de develop
#    → Railway despliega staging de develop

# 5. Validar en staging que todo funciona correctamente

# 6. Abrir PR en GitHub: develop → main
#    El CI vuelve a ejecutarse (es el código ya validado)
#    Si el CI está verde y hay aprobación → Squash and merge

# 7. main se actualiza automáticamente
#    → Vercel despliega a producción
#    → Railway despliega a producción
```

### 6.2 Hotfix urgente (bug en producción)

```bash
# 1. Crear hotfix desde main
git checkout main
git pull origin main
git checkout -b hotfix/descripcion-del-bug

# 2. Corregir el bug y subir
git add .
git commit -m "fix(scope): descripción del fix"
git push origin hotfix/descripcion-del-bug

# 3. Abrir PR urgente: hotfix/* → main
#    El CI ejecuta todos los checks
#    Si el CI está verde → Squash and merge

# 4. main se actualiza → deploy automático a producción

# 5. IMPORTANTE: propagar el fix a develop
git checkout develop
git pull origin develop
git merge main
git push origin develop
```

### 6.3 Tiempos estimados de despliegue

| Fase | Duración aproximada |
|------|-------------------|
| CI completo (lint + test + build + SonarCloud) | 5–10 minutos |
| Vercel build y deploy | 1–3 minutos |
| Railway build y deploy | 3–6 minutos |
| **Total desde merge hasta producción** | **~10–20 minutos** |

## 8. Rollback

### Rollback del Frontend (Vercel)

Vercel conserva todos los deploys anteriores. Para revertir:

1. Ir a **Vercel → Deployments**.
2. Localizar el último deploy estable.
3. Hacer clic en los tres puntos (**···**) → **Promote to Production**.
4. El deploy anterior queda activo en producción en segundos.

### Rollback del Backend (Railway)

Railway también conserva deploys anteriores:

1. Ir a **Railway → Deployments**.
2. Localizar el último deploy estable.
3. Hacer clic en **Redeploy** sobre ese deploy.

Como alternativa, revertir el commit en git y hacer push a `main`:

```bash
# Revertir el último commit de main (crea un nuevo commit de reversión)
git checkout main
git revert HEAD --no-edit
git push origin main
# → El CI se ejecuta y Railway despliega la versión revertida
```

## 9. Monitorización

### Logs en Railway

Ir a **Railway → Servicio → Logs** para ver los logs en tiempo real del backend (.NET con Serilog).

Los logs incluyen:
- Peticiones HTTP entrantes y sus tiempos de respuesta
- Errores y excepciones con stack trace

Ir a **Vercel → Functions** para ver los logs de las funciones serverless de Next.js (Server Components, API routes).

Ir a **GitHub → Actions** para ver el historial de ejecuciones del CI:
- Estado de cada job (verde/rojo)
- Logs detallados de cada paso

Acceder a [sonarcloud.io](https://sonarcloud.io) para:
- Ver el historial de análisis por rama y PR
- Comprobar métricas de cobertura de tests
- Revisar nuevos bugs, code smells o vulnerabilidades introducidos en cada PR

## 10. Troubleshooting

### El CI falla en `backend-lint`

**Causa:** El código no cumple el formato de `dotnet format`.

**Solución:**
```bash
cd backend
dotnet format
git add .
git commit -m "style: fix dotnet format issues"
git push
```

### El CI falla en `frontend-lint`

**Causa:** Errores de ESLint o TypeScript.

**Solución:**
```bash
cd frontend
npm run lint       # ver errores de ESLint
npm run type-check # ver errores de TypeScript
# Corregir y volver a pushear
```

### El CI falla en `frontend-build`

**Causa frecuente:** La variable `NEXT_PUBLIC_API_URL` no está definida como Variable de repositorio en GitHub y el fallback `http://localhost:5000/api` causa un error durante el build.

**Solución:** Definir `NEXT_PUBLIC_API_URL` en **GitHub → Settings → Variables → Actions**.

### Railway: la aplicación arranca pero la BD no conecta

**Causa:** `ConnectionStrings__DefaultConnection` incorrecto o PostgreSQL no disponible.

**Solución:**
1. Verificar en Railway → PostgreSQL → Variables que el servicio está corriendo.
2. Copiar el valor actualizado de `DATABASE_URL` y actualizar la variable del backend.
3. Railway hará redeploy automático al guardar la variable.

### Railway: error "No admin user exists and AdminSeed settings are not configured"

**Causa:** Primer arranque sin las variables `AdminSeed__Email` y `AdminSeed__Password`.

**Solución:** Añadir las variables en Railway → Servicio → Variables y Railway redesplegará automáticamente.

### Vercel: el frontend no puede conectar con el backend (CORS error)

**Causa:** La URL del frontend no está en `Cors__AllowedOrigins` del backend.

**Solución:** Añadir la URL exacta de Vercel (con protocolo, sin barra final) en las variables de Railway:
```
Cors__AllowedOrigins__0=https://simracingshop.com
Cors__AllowedOrigins__1=https://www.simracingshop.com
```

### SonarCloud no analiza el proyecto

**Causa:** El secret `SONAR_TOKEN` o la variable `SONAR_ORGANIZATION` no están configurados en GitHub.

**Solución:** Verificar en **GitHub → Settings → Secrets and variables → Actions** que ambos valores existen y son correctos.
