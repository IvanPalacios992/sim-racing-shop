## UC-AUTH-01: Registro de usuario

**Actores:** Usuario no autenticado

**Descripción:** Un visitante crea una nueva cuenta proporcionando sus datos personales y credenciales, obteniendo acceso inmediato al sistema.

### Precondiciones
- El usuario no tiene una sesión activa.
- El usuario se encuentra en la página de registro (`/[locale]/register`).

### Postcondiciones
- Se crea una cuenta con el rol "Customer".
- Se genera un token JWT y un refresh token.
- El usuario es redirigido a la página principal con sesión iniciada.

### Flujo principal
1. El sistema muestra el formulario de registro con los campos: nombre, apellido, email, contraseña, confirmar contraseña y aceptación de términos.
2. El usuario completa todos los campos obligatorios.
3. El sistema valida en el cliente:
   - Email con formato válido.
   - Contraseña con mínimo 8 caracteres, al menos una mayúscula, una minúscula y un dígito.
   - Contraseña y confirmación coinciden.
   - Términos aceptados.
4. El sistema envía la petición `POST /api/auth/register`.
5. El backend crea el usuario, le asigna el rol "Customer" y genera los tokens.
6. El sistema almacena los tokens y datos del usuario en el store local.
7. El sistema redirige al usuario a la página principal, mostrando el mensaje de bienvenida.

### Flujos alternativos

**FA-01: Email ya registrado**
- En el paso 5, si el email ya existe, el backend responde con `400 Bad Request`.
- El sistema muestra el mensaje "This email is already registered".
- El usuario puede corregir el email y reintentar.

**FA-02: Contraseña no cumple la política**
- En el paso 5, si la contraseña no cumple los requisitos del servidor, el backend responde con `400 Bad Request`.
- El sistema muestra el error devuelto por el servidor.

**FA-03: Validación del cliente falla**
- En el paso 3, si algún campo no es válido, el sistema muestra los errores inline junto a cada campo.
- No se realiza la petición al servidor.

**FA-04: Error de red**
- En el paso 4, si la petición falla por problemas de conexión, el sistema muestra un mensaje genérico de error.

## UC-AUTH-02: Inicio de sesión

**Actores:** Usuario no autenticado (registrado previamente)

**Descripción:** Un usuario existente accede al sistema proporcionando sus credenciales, obteniendo un token de acceso y un refresh token.

### Precondiciones
- El usuario tiene una cuenta registrada.
- El usuario no tiene una sesión activa.
- El usuario se encuentra en la página de login (`/[locale]/login`).

### Postcondiciones
- Se genera un token JWT (60 min) y un refresh token (7 días).
- El usuario es redirigido a la página principal con sesión iniciada.

### Flujo principal
1. El sistema muestra el formulario de login con los campos: email, contraseña, y la opción "Recordarme".
2. El usuario introduce su email y contraseña.
3. El sistema valida que el email tenga formato válido y la contraseña no esté vacía.
4. El sistema envía la petición `POST /api/auth/login`.
5. El backend verifica las credenciales y genera los tokens.
6. El sistema almacena los tokens y datos del usuario en el store local.
7. El sistema redirige al usuario a la página principal.

### Flujos alternativos

**FA-01: Credenciales inválidas**
- En el paso 5, si el email no existe o la contraseña es incorrecta, el backend responde con `401 Unauthorized`.
- El sistema muestra "Invalid email or password. Please try again."
- El contador de intentos fallidos se incrementa en 1.

**FA-02: Cuenta bloqueada**
- En el paso 5, si el usuario ha alcanzado 5 intentos fallidos consecutivos, el backend responde con `401 Unauthorized` indicando bloqueo temporal.
- El sistema muestra "Account temporarily locked. Please try again later."
- La cuenta se desbloquea automáticamente tras 15 minutos.

**FA-03: Email no verificado**
- En el paso 5, si la verificación de email está habilitada y el email no ha sido verificado, el backend responde con `403 Forbidden`.
- El sistema muestra "Please verify your email before signing in."

**FA-04: Usuario autenticado accede al login**
- En la precondición, si el usuario ya tiene sesión activa, el `AuthRedirectGuard` lo redirige automáticamente a la página principal.
- No se muestra el formulario de login.

## UC-AUTH-03: Cierre de sesión

**Actores:** Usuario autenticado

**Descripción:** Un usuario con sesión activa cierra su sesión, revocando todos sus tokens e invalidando el acceso.

### Precondiciones
- El usuario tiene una sesión activa con tokens válidos.

### Postcondiciones
- Todos los refresh tokens del usuario son revocados.
- El security stamp se actualiza, invalidando todos los JWT existentes.
- El estado local de autenticación se limpia.
- El usuario ve la interfaz de visitante (botones "Sign In" y "Create Account").

### Flujo principal
1. El usuario hace clic en el botón "Log Out" visible en la página principal.
2. El sistema envía la petición `POST /api/auth/logout` con el token JWT.
3. El backend revoca todos los refresh tokens activos del usuario.
4. El backend actualiza el security stamp del usuario.
5. El sistema limpia el store local (token, usuario, estado de autenticación).
6. La interfaz se actualiza mostrando los enlaces de "Sign In" y "Create Account".

### Flujos alternativos

**FA-01: Error en la petición de logout**
- En el paso 2, si la petición falla (error de red, token expirado), el sistema limpia igualmente el estado local.
- El usuario queda desconectado en el frontend independientemente de la respuesta del servidor.

## UC-AUTH-04: Renovación de token

**Actores:** Sistema (automático)

**Descripción:** El sistema renueva automáticamente el token JWT cuando una petición recibe un error 401, utilizando el refresh token para obtener nuevos credenciales sin intervención del usuario.

### Precondiciones
- El usuario tiene una sesión activa.
- Existe un refresh token válido (no expirado, no revocado) en el store local.

### Postcondiciones
- Se genera un nuevo token JWT y un nuevo refresh token.
- El refresh token anterior es revocado y reemplazado (rotación de tokens).
- La petición original que falló se reintenta con el nuevo token.

### Flujo principal
1. El cliente HTTP intercepta una respuesta `401 Unauthorized` en cualquier petición autenticada.
2. El interceptor envía la petición `POST /api/auth/refresh-token` con el refresh token actual.
3. El backend valida el refresh token, lo revoca y genera un par de tokens nuevos.
4. El sistema actualiza el store local con los nuevos tokens.
5. El interceptor reenvía la petición original con el nuevo token JWT.
6. El usuario continúa su actividad sin interrupciones.

### Flujos alternativos

**FA-01: Refresh token expirado o revocado**
- En el paso 3, si el refresh token no es válido, el backend responde con `401 Unauthorized`.
- El sistema limpia el estado de autenticación.
- El usuario es redirigido al login.

**FA-02: Múltiples peticiones concurrentes fallan**
- En el paso 1, si varias peticiones reciben 401 simultáneamente, el interceptor encola las peticiones pendientes.
- Solo se realiza una petición de refresh.
- Una vez completada la renovación, todas las peticiones encoladas se reintentan con el nuevo token.

## UC-AUTH-05: Solicitud de restablecimiento de contraseña

**Actores:** Usuario no autenticado

**Descripción:** Un usuario que ha olvidado su contraseña solicita un enlace de restablecimiento por email. El sistema responde siempre con éxito por seguridad, independientemente de si el email existe.

### Precondiciones
- El usuario se encuentra en la página de "Forgot Password" (`/[locale]/forgot-password`).

### Postcondiciones
- Si el email existe, se envía un correo con un enlace de restablecimiento (token válido por 24 horas).
- El sistema muestra la pantalla de confirmación con la dirección de email indicada.

### Flujo principal
1. El sistema muestra el formulario con un campo de email.
2. El usuario introduce su dirección de email.
3. El sistema valida que el formato del email sea correcto.
4. El sistema envía la petición `POST /api/auth/forgot-password`.
5. El backend genera un token de restablecimiento y envía el email (si la cuenta existe).
6. El backend responde siempre con `200 OK` (prevención de enumeración de emails).
7. El sistema muestra la pantalla de confirmación "Check Your Email" con la dirección proporcionada.
8. Se inicia un temporizador de 60 segundos para permitir reenvío.

### Flujos alternativos

**FA-01: Email no registrado**
- En el paso 5, si el email no existe, el backend no envía correo pero responde igualmente con `200 OK`.
- El usuario ve la misma pantalla de confirmación (no se revela si el email existe).

**FA-02: Reenvío de email**
- Tras el paso 8, una vez transcurridos los 60 segundos, el botón "Resend Email" se habilita.
- El usuario puede hacer clic para reenviar la solicitud.
- Se reinicia el temporizador de 60 segundos.

**FA-03: Cambiar email**
- Desde la pantalla de confirmación, el usuario hace clic en "Try a different email".
- El sistema vuelve a mostrar el formulario con el campo de email vacío.

**FA-04: Error de red**
- En el paso 4, si la petición falla, el sistema muestra "Could not send email. Please try again."

## UC-AUTH-06: Restablecimiento de contraseña

**Actores:** Usuario no autenticado

**Descripción:** Un usuario establece una nueva contraseña utilizando el enlace de restablecimiento recibido por email, que contiene un token de un solo uso.

### Precondiciones
- El usuario ha recibido un email con el enlace de restablecimiento.
- El token no ha expirado (24 horas desde su generación) y no ha sido utilizado.
- El usuario accede a `/[locale]/reset-password?token={token}&email={email}`.

### Postcondiciones
- La contraseña del usuario se actualiza.
- El security stamp se actualiza, invalidando todos los tokens JWT y refresh tokens existentes.
- El usuario debe iniciar sesión nuevamente con la nueva contraseña.

### Flujo principal
1. El sistema muestra el formulario con los campos: nueva contraseña y confirmar contraseña, junto con los indicadores de requisitos de contraseña.
2. El usuario introduce la nueva contraseña cumpliendo los requisitos:
   - Mínimo 8 caracteres.
   - Al menos una letra mayúscula.
   - Al menos una letra minúscula.
   - Al menos un dígito.
3. El usuario confirma la contraseña.
4. El sistema valida que ambas contraseñas coincidan y cumplan los requisitos.
5. El sistema envía la petición `POST /api/auth/reset-password` con el email, token y nueva contraseña.
6. El backend verifica el token, actualiza la contraseña, revoca todos los tokens activos y actualiza el security stamp.
7. El sistema muestra la pantalla de éxito con un enlace para iniciar sesión.

### Flujos alternativos

**FA-01: Token inválido o expirado**
- En el paso 6, si el token no es válido o ha expirado, el backend responde con `400 Bad Request`.
- El sistema muestra la pantalla de "token inválido" con un enlace a "Forgot Password" para solicitar uno nuevo.

**FA-02: Contraseña no cumple requisitos**
- En el paso 4, si la contraseña no cumple los requisitos, el sistema muestra los indicadores de requisitos no cumplidos.
- No se realiza la petición al servidor.

**FA-03: Contraseñas no coinciden**
- En el paso 4, si las contraseñas no coinciden, el sistema muestra el error "Passwords do not match".

## UC-AUTH-07: Consulta de perfil de usuario

**Actores:** Usuario autenticado

**Descripción:** El sistema obtiene los datos del perfil del usuario autenticado para mostrar información personalizada en la interfaz.

### Precondiciones
- El usuario tiene una sesión activa con un token JWT válido.

### Postcondiciones
- Se obtienen los datos actualizados del usuario (id, email, nombre, apellido, idioma, estado de verificación, roles).

### Flujo principal
1. El sistema envía la petición `GET /api/auth/me` con el token JWT en la cabecera de autorización.
2. El backend valida el token JWT y el security stamp.
3. El backend busca al usuario por ID y obtiene sus roles.
4. El backend responde con los datos del usuario (`UserDto`).
5. El sistema actualiza el store local con la información recibida.

### Flujos alternativos

**FA-01: Token expirado**
- En el paso 2, si el token JWT ha expirado, el backend responde con `401 Unauthorized`.
- Se activa el flujo de renovación de token (UC-AUTH-04).
- Tras la renovación, se reintenta la petición.

**FA-02: Usuario no encontrado**
- En el paso 3, si el usuario ya no existe en el sistema, el backend responde con `404 Not Found`.
- El sistema limpia el estado de autenticación y redirige al login.

**FA-03: Security stamp inválido**
- En el paso 2, si el security stamp del token no coincide con el actual (por cambio de contraseña o logout en otro dispositivo), el backend rechaza el token.
- Se limpia el estado de autenticación y se redirige al login.

## UC-CAT-01: Explorar catálogo de productos

**Actores:** Visitante, Usuario autenticado

**Descripción:** Un usuario navega por el listado paginado de productos, pudiendo filtrar por categoría, buscar por texto y ordenar los resultados. La respuesta está localizada según el idioma seleccionado.

### Precondiciones
- El usuario se encuentra en la página de catálogo (`/[locale]/productos`).

### Postcondiciones
- Se muestra el listado de productos que cumple los criterios de búsqueda y filtrado, paginado.

### Flujo principal
1. El sistema envía la petición `GET /api/products` con los parámetros de paginación y locale.
2. El backend devuelve la primera página de productos activos con nombre, imagen principal, precio base y slug.
3. El sistema muestra la cuadrícula de productos con el número total de resultados.

### Flujos alternativos

**FA-01: Filtrado por categoría o texto**
- El usuario selecciona una categoría o escribe en el buscador.
- El sistema envía `GET /api/products?search=...&categoryId=...` con los filtros aplicados.
- Se muestra la lista filtrada y el contador de resultados se actualiza.

**FA-02: Sin resultados**
- Si el backend devuelve una lista vacía, el sistema muestra el estado vacío "No se encontraron productos".

**FA-03: Navegación entre páginas**
- El usuario hace clic en un número de página o en "Siguiente".
- El sistema envía la petición con el parámetro `page` actualizado y reemplaza el listado actual.

## UC-CAT-02: Ver detalle de producto

**Actores:** Visitante, Usuario autenticado

**Descripción:** Un usuario accede a la página de detalle de un producto para ver su información completa: nombre, descripción, precio, imágenes, especificaciones técnicas y opciones de componentes para personalización 3D.

### Precondiciones
- El usuario accede a la URL del producto (`/[locale]/productos/[slug]`) o hace clic en una tarjeta del catálogo.

### Postcondiciones
- Se muestra la información completa del producto en el idioma seleccionado.

### Flujo principal
1. El sistema envía `GET /api/products/slug/{slug}?locale={locale}`.
2. El backend devuelve el detalle del producto: traducciones, imágenes ordenadas, especificaciones técnicas y opciones de componentes agrupadas por grupo de opción.
3. El sistema muestra la galería de imágenes, descripción, precio con IVA calculado y las opciones de personalización.
4. Si el producto tiene opciones de componentes (`IsCustomizable = true`), se renderiza el configurador 3D con el visor del modelo `.glb`.

### Flujos alternativos

**FA-01: Producto no encontrado**
- Si el backend responde con `404 Not Found`, el sistema muestra la página 404.

**FA-02: Cambio de opción en el configurador**
- El usuario selecciona una opción distinta en un grupo del configurador (ej. color del grip).
- El visor 3D intercambia el objeto 3D correspondiente (`GlbObjectName`) en tiempo real.
- El precio total se recalcula sumando los modificadores de precio de las opciones seleccionadas.

## UC-CART-01: Ver carrito

**Actores:** Visitante (con sesión anónima), Usuario autenticado

**Descripción:** El usuario consulta el contenido actual de su carrito, que incluye los productos añadidos, sus cantidades, precios unitarios, subtotal, IVA y coste de envío estimado.

### Precondiciones
- El carrito puede existir en dos contextos:
  - **Anónimo:** se identifica mediante la cabecera `X-Cart-Session: {uuid}` generada en el cliente.
  - **Autenticado:** se identifica por el `userId` del token JWT.

### Postcondiciones
- Se devuelve el contenido actual del carrito con los totales calculados.

### Flujo principal
1. El sistema envía `GET /api/cart?locale={locale}` con la cabecera de sesión o el token JWT.
2. El backend resuelve la clave del carrito (`cart:user:{userId}` o `cart:session:{sessionId}`) en Redis.
3. El backend enriquece cada item con los datos actualizados del producto (nombre, imagen, precio) y calcula subtotales.
4. El sistema muestra el listado de items y el resumen de precios.

### Flujos alternativos

**FA-01: Carrito vacío**
- Si el carrito no contiene items, el sistema muestra el estado vacío con un enlace al catálogo.

**FA-02: Sin identificación**
- Si la petición no incluye token JWT ni cabecera `X-Cart-Session`, el backend responde con `400 Bad Request`.

## UC-CART-02: Añadir producto al carrito

**Actores:** Visitante (con sesión anónima), Usuario autenticado

**Descripción:** El usuario añade un producto al carrito. Si el producto ya existe, se incrementa su cantidad. Si el producto no existe, se añade como nueva línea.

### Precondiciones
- El usuario tiene una sesión anónima o autenticada.
- El producto existe y está activo (`IsActive = true`).

### Postcondiciones
- El producto aparece en el carrito con la cantidad indicada.
- Si el producto ya estaba, la cantidad se incrementa en el valor especificado.
- Se devuelve el carrito actualizado con los totales recalculados.

### Flujo principal
1. El usuario hace clic en "Añadir al carrito" en la página de producto o en el configurador.
2. El sistema envía `POST /api/cart/items` con `{ productId, quantity }` y la cabecera de sesión o JWT.
3. El backend verifica que el producto existe en la base de datos.
4. El backend actualiza la entrada del carrito en Redis y recalcula totales.
5. El sistema actualiza el mini carrito en la barra de navegación y muestra un toast de confirmación.

### Flujos alternativos

**FA-01: Producto no encontrado**
- En el paso 3, si el producto no existe o está inactivo, el backend responde con `404 Not Found`.
- El sistema muestra un mensaje de error.

**FA-02: Cantidad inválida**
- Si la cantidad enviada es menor que 1 o mayor que 99, la validación del servidor rechaza la petición con `400 Bad Request`.

---

## UC-CART-03: Actualizar cantidad de un producto

**Actores:** Visitante (con sesión anónima), Usuario autenticado

**Descripción:** El usuario modifica la cantidad de un producto ya existente en el carrito.

### Precondiciones
- El producto con el `productId` indicado existe en el carrito activo.

### Postcondiciones
- La cantidad del producto se actualiza al valor indicado.
- Los totales del carrito se recalculan.

### Flujo principal
1. El usuario cambia el selector de cantidad de un item en el carrito.
2. El sistema envía `PUT /api/cart/items/{productId}` con `{ quantity }`.
3. El backend actualiza la cantidad en Redis y recalcula los totales.
4. El sistema refresca el resumen del carrito.

### Flujos alternativos

**FA-01: Producto no en el carrito**
- Si el `productId` no existe en el carrito, el backend responde con `404 Not Found`.

**FA-02: Cantidad fuera de rango**
- Si la cantidad no está entre 1 y 99, el backend responde con `400 Bad Request`.

---

## UC-CART-04: Eliminar producto del carrito

**Actores:** Visitante (con sesión anónima), Usuario autenticado

**Descripción:** El usuario elimina un producto concreto del carrito.

### Precondiciones
- El producto con el `productId` indicado existe en el carrito activo.

### Postcondiciones
- El producto queda eliminado del carrito.
- Los totales se recalculan.

### Flujo principal
1. El usuario hace clic en el icono de eliminar junto a un item.
2. El sistema envía `DELETE /api/cart/items/{productId}`.
3. El backend elimina la entrada del producto en Redis.
4. El sistema actualiza el carrito y el contador del mini carrito.

---

## UC-CART-05: Vaciar carrito

**Actores:** Visitante (con sesión anónima), Usuario autenticado

**Descripción:** El usuario elimina todos los productos del carrito de una sola vez.

### Precondiciones
- El carrito tiene al menos un producto.

### Postcondiciones
- El carrito queda completamente vacío.

### Flujo principal
1. El usuario hace clic en "Vaciar carrito".
2. El sistema envía `DELETE /api/cart`.
3. El backend elimina la clave completa del carrito en Redis.
4. El sistema muestra el estado vacío del carrito.

---

## UC-CART-06: Fusionar carrito anónimo tras inicio de sesión

**Actores:** Sistema (automático tras UC-AUTH-02)

**Descripción:** Al iniciar sesión, el sistema fusiona automáticamente el carrito de la sesión anónima con el carrito del usuario autenticado. Si hay productos repetidos, las cantidades se suman. El carrito de sesión queda eliminado tras la fusión.

### Precondiciones
- El usuario acaba de completar el inicio de sesión (UC-AUTH-02).
- Existe un `sessionId` de carrito anónimo almacenado en el cliente.

### Postcondiciones
- El carrito del usuario contiene todos los productos del carrito anónimo y del carrito autenticado previo.
- El carrito de sesión anónima queda eliminado en Redis.

### Flujo principal
1. Tras el login, el sistema detecta que existe un `sessionId` de carrito anónimo en el cliente.
2. El sistema envía `POST /api/cart/merge` con `{ sessionId }` y el nuevo token JWT.
3. El backend lee ambos carritos (sesión y usuario) de Redis.
4. El backend combina los items: si un producto existe en ambos, suma las cantidades.
5. El backend guarda el carrito fusionado bajo `cart:user:{userId}` y elimina `cart:session:{sessionId}`.
6. El sistema elimina el `sessionId` del cliente y actualiza el mini carrito.

### Flujos alternativos

**FA-01: Carrito anónimo vacío**
- Si no había productos en la sesión anónima, la fusión no modifica el carrito del usuario.

**FA-02: Sin carrito previo del usuario**
- El carrito del usuario queda igual al carrito anónimo.

## UC-ORD-01: Crear pedido

**Actores:** Usuario autenticado

**Descripción:** El usuario formaliza la compra con los productos del carrito, una dirección de envío y el método de pago. El sistema genera el pedido, calcula el coste de envío según la zona y envía una confirmación por email.

### Precondiciones
- El usuario está autenticado.
- El carrito contiene al menos un producto.
- El usuario ha proporcionado una dirección de envío válida.

### Postcondiciones
- Se crea el pedido con estado `pending` y un número único (`ORD-{YEAR}-{SEQUENTIAL}`).
- Se genera un registro de pago asociado al pedido.
- El usuario recibe un email de confirmación con el resumen del pedido.
- El carrito queda vacío.

### Flujo principal
1. El usuario revisa el resumen del pedido en el checkout y confirma la dirección de envío.
2. El sistema envía `POST /api/shipping/calculate` con el código postal y el peso total para obtener el coste de envío.
3. El sistema presenta el desglose final: subtotal, IVA (21%), coste de envío y total.
4. El usuario confirma y el sistema envía `POST /api/orders` con los datos del pedido.
5. El backend valida la dirección de envío y calcula los totales.
6. El backend genera el número de pedido y crea los registros en base de datos (`Orders` y `OrderItems` con snapshot de datos del producto).
7. El backend envía el email de confirmación de forma asíncrona (fire-and-forget).
8. El sistema redirige al usuario a la página de detalle del pedido recién creado.

### Flujos alternativos

**FA-01: Dirección de envío inválida**
- En el paso 5, si la dirección no cumple las validaciones, el backend responde con `400 Bad Request` y el mensaje de error correspondiente.

**FA-02: Zona de envío no encontrada**
- Si el código postal no corresponde a ninguna zona de envío activa, el backend responde con `400 Bad Request`.
- El sistema muestra el error e invita al usuario a comprobar el código postal.

**FA-03: Error en el envío del email**
- El envío de email falla silenciosamente (fire-and-forget): el pedido se crea con éxito independientemente.

## UC-ORD-02: Consultar historial de pedidos

**Actores:** Usuario autenticado

**Descripción:** El usuario consulta la lista de todos sus pedidos, ordenados por fecha de creación descendente.

### Precondiciones
- El usuario está autenticado.

### Postcondiciones
- Se devuelve la lista de pedidos del usuario con su estado y resumen de items.

### Flujo principal
1. El usuario accede a "Mis pedidos" en el área privada.
2. El sistema envía `GET /api/orders` con el token JWT.
3. El backend filtra los pedidos por `userId` extraído del token.
4. El sistema muestra las tarjetas de pedido con número, estado, total y fecha.

### Flujos alternativos

**FA-01: Sin pedidos**
- Si el usuario no tiene pedidos, el sistema muestra el estado vacío "Aún no tienes pedidos".

## UC-ORD-03: Ver detalle de un pedido

**Actores:** Usuario autenticado

**Descripción:** El usuario consulta el detalle completo de un pedido concreto: items, dirección de envío, desglose de precios, estado actual y número de seguimiento (si está disponible).

### Precondiciones
- El usuario está autenticado.
- El pedido pertenece al usuario.

### Postcondiciones
- Se muestra el detalle completo del pedido.

### Flujo principal
1. El usuario hace clic en un pedido de su historial.
2. El sistema envía `GET /api/orders/{id}` con el token JWT.
3. El backend verifica que el `userId` del pedido coincide con el del token.
4. El sistema muestra el detalle: items con snapshot de configuración, dirección de envío, desglose de precios y estado.

### Flujos alternativos

**FA-01: Pedido no encontrado**
- Si el pedido no existe, el backend responde con `404 Not Found`.

**FA-02: Pedido de otro usuario**
- Si el `userId` del pedido no coincide con el del token, el backend responde con `401 Unauthorized`.
- El sistema redirige al historial de pedidos.

## UC-USER-01: Editar datos personales

**Actores:** Usuario autenticado

**Descripción:** El usuario modifica su nombre, apellido o dirección de email desde la sección de perfil.

### Precondiciones
- El usuario está autenticado.

### Postcondiciones
- Los datos del usuario se actualizan en base de datos.
- El store local se actualiza con los nuevos valores.

### Flujo principal
1. El usuario accede a "Mi perfil" y hace clic en "Editar".
2. El sistema muestra el modal con los campos: nombre, apellido y email pre-rellenados.
3. El usuario modifica los campos deseados y confirma.
4. El sistema envía `PUT /api/user` con los nuevos valores.
5. El backend actualiza el usuario en base de datos y devuelve el `UserDetailDto` actualizado.
6. El sistema cierra el modal y actualiza la interfaz con los nuevos datos.

### Flujos alternativos

**FA-01: Usuario no encontrado**
- Si el usuario ya no existe (cuenta eliminada en otra sesión), el backend responde con `404 Not Found`.
- El sistema limpia la sesión y redirige al login.

## UC-USER-02: Gestionar dirección de facturación

**Actores:** Usuario autenticado

**Descripción:** El usuario crea o actualiza su única dirección de facturación. Cada usuario tiene como máximo una dirección de facturación.

### Precondiciones
- El usuario está autenticado.

### Postcondiciones
- La dirección de facturación queda creada o actualizada.

### Flujo principal — Crear (primera vez)
1. El usuario accede a "Mi perfil → Direcciones" y hace clic en "Añadir dirección de facturación".
2. El sistema muestra el modal con los campos: calle, ciudad, provincia, código postal y país.
3. El usuario rellena el formulario y confirma.
4. El sistema envía `POST /api/addresses/billing` con los datos y el `userId`.
5. El backend crea la dirección con `AddressType = Billing` e `IsDefault = true`.
6. El sistema muestra la nueva dirección en el perfil.

### Flujo principal — Editar (existente)
1. El usuario hace clic en "Editar" sobre su dirección de facturación.
2. El sistema muestra el modal con los datos actuales.
3. El usuario modifica los campos y confirma.
4. El sistema envía `PUT /api/addresses/billing` con los nuevos valores.
5. El backend actualiza la dirección existente.

### Flujos alternativos

**FA-01: Dirección no encontrada al editar**
- Si la dirección no existe al intentar actualizarla, el backend responde con `404 Not Found`.

## UC-SHIP-01: Calcular coste de envío

**Actores:** Sistema (invocado desde UC-ORD-01 o desde el checkout)

**Descripción:** El sistema calcula el coste de envío para un código postal, subtotal de pedido y peso total dados, determinando si aplica envío gratuito.

### Precondiciones
- Existe al menos una zona de envío activa configurada en el sistema.
- El código postal proporcionado tiene un formato válido.

### Postcondiciones
- Se devuelve el coste de envío, el nombre de la zona y si aplica envío gratuito.

### Flujo principal
1. El sistema envía `POST /api/shipping/calculate` con `{ postalCode, subtotal, weightKg }`.
2. El backend identifica la zona de envío cuyo prefijo de código postal coincide con el proporcionado.
3. El backend calcula el coste: `BaseCost + (weightKg × CostPerKg)`.
4. Si `subtotal >= FreeShippingThreshold`, el coste de envío es 0.
5. El sistema muestra el coste calculado y el umbral de envío gratuito restante (si aplica).

### Flujos alternativos

**FA-01: Código postal sin zona asignada**
- Si ninguna zona activa cubre el prefijo del código postal, el backend responde con `400 Bad Request`.
- El sistema informa al usuario de que no hay envío disponible para esa zona.

## UC-ADMIN-01: Listar todos los pedidos

**Actores:** Administrador

**Descripción:** El administrador consulta el listado paginado de todos los pedidos del sistema, con capacidad de filtrar por estado y buscar por número de pedido o email del cliente.

### Precondiciones
- El usuario tiene el rol `Admin`.

### Postcondiciones
- Se devuelve la página de pedidos filtrada con el conteo total.

### Flujo principal
1. El administrador accede al panel de administración → Pedidos.
2. El sistema envía `GET /api/admin/orders?page=1&pageSize=20` con el token JWT de admin.
3. El backend verifica el rol `Admin` mediante `[Authorize(Roles = "Admin")]`.
4. El backend devuelve la lista paginada con: número de pedido, email del cliente, total, estado, fecha y número de items.
5. El sistema muestra la tabla de pedidos con controles de paginación.

### Flujos alternativos

**FA-01: Filtrar por estado**
- El administrador selecciona un estado en el desplegable.
- El sistema envía `GET /api/admin/orders?status=pending`.
- La tabla se actualiza con los pedidos del estado seleccionado.

**FA-02: Buscar por número o email**
- El administrador escribe en el buscador.
- El sistema envía `GET /api/admin/orders?search=ORD-2026` o `?search=cliente@email.com`.

**FA-03: Sin permisos**
- Si el token no tiene el rol `Admin`, el backend responde con `403 Forbidden`.

## UC-ADMIN-03: Avanzar estado de un pedido

**Actores:** Administrador

**Descripción:** El administrador avanza el estado de un pedido siguiendo el flujo secuencial obligatorio. Solo se permite la transición al estado inmediatamente siguiente; los saltos de estado no están permitidos.

### Precondiciones
- El usuario tiene el rol `Admin`.
- El pedido existe y no está en un estado terminal (`delivered` o `cancelled`).

### Postcondiciones
- El estado del pedido avanza al siguiente estado permitido.

### Flujo de estados
```
pending → processing → shipped → delivered
```
> Los estados `delivered` y `cancelled` son terminales. No se puede retroceder de estado.

### Flujo principal
1. El administrador hace clic en "Avanzar estado" en el detalle del pedido.
2. El sistema envía `PATCH /api/admin/orders/{id}/status` con `{ status: "<siguiente estado>" }`.
3. El backend valida que la transición es la correcta según el diccionario de transiciones válidas.
4. El backend actualiza el `OrderStatus` del pedido en base de datos.
5. El sistema actualiza el estado mostrado en el modal y en la tabla de pedidos.

### Flujos alternativos

**FA-01: Transición no válida**
- Si el `status` enviado no es el siguiente estado esperado, el backend responde con `400 Bad Request` indicando cuál es el estado correcto.
- Por ejemplo: intentar pasar de `pending` a `shipped` directamente devuelve: *"Transición no válida de 'pending' a 'shipped'. El siguiente estado debe ser 'processing'."*

**FA-02: Estado terminal**
- Si el pedido ya está en `delivered` o `cancelled`, el backend responde con `400 Bad Request`: *"El pedido ya está en estado terminal"*.

**FA-03: Pedido no encontrado**
- El backend responde con `404 Not Found`.
