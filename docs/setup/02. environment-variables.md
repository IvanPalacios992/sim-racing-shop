# Variables de Entorno

## Descripción General

Este documento detalla todas las variables de entorno necesarias para el proyecto, organizadas por servicio y entorno.

**IMPORTANTE:**
- **NUNCA** commitear archivos con valores reales (`.env.local`, `appsettings.Development.json`)
- Usar `.env.example` y `appsettings.example.json` como plantillas
- Valores sensibles en producción deben estar en servicios seguros (Vercel Secrets, Railway Env Vars)

---

## Frontend (Next.js)

### Archivos de Configuración

```
frontend/
├── .env.example          # Plantilla para compartir (commiteable)
└── .env.local            # Local development (NO commitear)
```

### Variables Públicas (NEXT_PUBLIC_*)

Estas variables son accesibles en el cliente (browser).

#### API y Backend

```bash
# URL del backend API
NEXT_PUBLIC_API_URL=https://localhost:7002/api

# Producción:
# NEXT_PUBLIC_API_URL=https://api.simracingshop.com/api
```

**Uso:** Todas las llamadas a la API desde el frontend
**Requerido:** ✅ Sí
**Valor local:** `https://localhost:7002/api`
**Valor producción:** URL pública del backend desplegado

> **Nota:** En desarrollo el backend corre en `https://localhost:7002` (Kestrel con HTTPS). En producción se usa el dominio real.

```bash
# ============================================

# ============================================
```

## Descripción Detallada de Variables Backend

### ConnectionStrings

```json
"ConnectionStrings": {
  "DefaultConnection": "Host=localhost;Database=simracing;Username=postgres;Password=postgres;Port=5432",
  "Redis": "localhost:6379"
}
```

**DefaultConnection:** Cadena de conexión a PostgreSQL
- `Host`: Servidor PostgreSQL
- `Database`: Nombre de la base de datos
- `Username` / `Password`: Credenciales
- `Port`: Puerto (default 5432)

**Producción:**
```json
"DefaultConnection": "Host=your-db.example.com;Database=simracing;Username=postgres;Password=xxx;Port=5432;SSL Mode=Require"
```

**Redis:** Connection string de Redis usado por StackExchange.Redis
- Local: `localhost:6379`
- Upstash/producción: `redis-xxxxx.upstash.io:6379,password=xxx,ssl=true`

---

### JwtSettings

```json
"JwtSettings": {
  "Secret": "your-super-secret-key-change-this-min-32-characters",
  "Issuer": "SimRacingShop",
  "Audience": "SimRacingShop",
  "ExpiryMinutes": 60,
  "RefreshTokenExpiryDays": 7
}
```

**Secret:** ⚠️ CRÍTICO — debe ser aleatorio y mínimo 32 caracteres
```bash
# Generar un secret seguro:
openssl rand -base64 32
```

**Issuer / Audience:** Identificadores del token (pueden ser iguales)

**ExpiryMinutes:** Duración del access token JWT (60 = 1 hora)

**RefreshTokenExpiryDays:** Duración del refresh token (7 días)

> **Nota:** La sección se llama `JwtSettings` en el código, no `Jwt`. Usar el nombre exacto al configurar variables de entorno.

---

### AdminSeed (Usuario Admin Inicial)

```json
"AdminSeed": {
  "Email": "admin@simracingshop.com",
  "Password": "TuPasswordSeguro123!",
  "FirstName": "Admin",
  "LastName": "SimRacing"
}
```

**⚠️ IMPORTANTE:** Estas variables son **requeridas en el primer arranque** si no existe ningún usuario admin en la base de datos. Si no están configuradas y no hay admin, la aplicación falla con error.

| Campo | Requerido | Descripción |
|-------|:---------:|-------------|
| `Email` | ✅ | Email de acceso del admin (debe ser válido) |
| `Password` | ✅ | Mín. 8 chars, mayúscula + minúscula + dígito |
| `FirstName` | ⚠️ Opcional | Default: `"Admin"` |
| `LastName` | ⚠️ Opcional | Default: `"SimRacing"` |

**Como variables de entorno (Railway/producción):**
```bash
AdminSeed__Email=admin@simracingshop.com
AdminSeed__Password=TuPasswordSeguro123!
AdminSeed__FirstName=Admin
AdminSeed__LastName=SimRacing
```

> Una vez creado el admin estas variables pueden quedar vacías. El seeder solo actúa si no existe ningún admin en el sistema.

---

### Stripe

```json
"Stripe": {
  "SecretKey": "sk_test_...",
  "WebhookSecret": "whsec_...",
  "Currency": "eur"
}
```

**SecretKey:** Clave secreta de Stripe
- Test: `sk_test_...`
- Live: `sk_live_...`

**WebhookSecret:** Secret para verificar eventos de Stripe webhook
- Obtener en Stripe Dashboard → Developers → Webhooks
- Puede quedar vacío si no se usan webhooks localmente

**Currency:** Divisa ISO 4217 (eur, usd, gbp…)

**Obtener claves:** [Stripe Dashboard → API keys](https://dashboard.stripe.com/apikeys)

---

### Redis (sección separada)

```json
"Redis": {
  "Configuration": "localhost:6379",
  "InstanceName": "SimRacing_"
}
```

> El proyecto tiene Redis configurado en dos lugares: `ConnectionStrings.Redis` (para StackExchange.Redis) y esta sección `Redis` (para el cliente con prefijo de instancia). Ambos deben apuntar al mismo servidor.

**Configuration:** Connection string completo
- Local: `localhost:6379`
- Producción: `redis-xxxxx.upstash.io:6379,password=xxx,ssl=true`

**InstanceName:** Prefijo para todas las keys de caché (evita colisiones entre entornos)

### ResendSettings (Email)

```json
"ResendSettings": {
  "ApiKey": "re_...",
  "FromEmail": "noreply@simracingshop.com",
  "FromName": "SimRacing Shop",
  "FrontendBaseUrl": "https://simracingshop.com"
}
```

**ApiKey:** Obtener en [Resend Dashboard → API Keys](https://resend.com/api-keys)

**FromEmail:** Dirección de origen
- Desarrollo: usar `onboarding@resend.dev` (funciona sin verificar dominio)
- Producción: dirección del dominio verificado en Resend

**FromName:** Nombre que aparece en los emails enviados

**FrontendBaseUrl:** URL base del frontend, usada para construir links en emails (verificación, reset de contraseña). Sin barra final.
- Local: `http://localhost:3000`
- Producción: `https://simracingshop.com`

### Serilog (Logging)

```json
"Serilog": {
  "MinimumLevel": {
    "Default": "Information",
    "Override": {
      "Microsoft": "Warning",
      "Microsoft.AspNetCore": "Warning",
      "Microsoft.EntityFrameworkCore": "Warning",
      "System": "Warning"
    }
  },
  "WriteTo": [
    {
      "Name": "Console",
      "Args": {
        "outputTemplate": "[{Timestamp:HH:mm:ss} {Level:u3}] {Message:lj} {Properties:j}{NewLine}{Exception}"
      }
    },
    {
      "Name": "File",
      "Args": {
        "path": "logs/log-.txt",
        "rollingInterval": "Day",
        "retainedFileCountLimit": 7,
        "shared": true
      }
    },
    {
      "Name": "Seq",
      "Args": { "serverUrl": "http://localhost:5341" }
    }
  ],
  "Enrich": ["FromLogContext", "WithMachineName", "WithThreadId", "WithProcessId"],
  "Properties": {
    "Application": "SimRacingShop.API"
  }
}
```

**MinimumLevel:** Niveles disponibles (de menor a mayor): `Verbose` → `Debug` → `Information` → `Warning` → `Error` → `Fatal`

**Sinks configurados:**
- **Console:** Salida estándar, siempre activa
- **File:** Archivos rotativos diarios en `logs/log-.txt`, retención 7 días
- **Seq:** UI de logs estructurados en `http://localhost:5341` (Docker local)

### Cors

```json
"Cors": {
  "AllowedOrigins": ["http://localhost:3000"]
}
```

**AllowedOrigins:** Lista de orígenes permitidos para CORS
- Desarrollo: `["http://localhost:3000"]`
- Producción: `["https://simracingshop.com"]`

⚠️ En producción incluir solo dominios específicos, nunca `*`

## Configuración en Servicios Cloud

### Vercel (Frontend)

**Vía UI:**
1. Dashboard → Tu Proyecto → Settings → Environment Variables
2. Añadir variables, elegir entornos: Production, Preview, Development

**Vía CLI:**
```bash
vercel env add NEXT_PUBLIC_API_URL production
vercel env ls
vercel env pull .env.local
```

**Notas:**
- Variables `NEXT_PUBLIC_*` se incluyen en el bundle del cliente
- Cambios requieren redeploy

---

### Railway (Backend)

**Vía UI:**
1. Dashboard → Tu Servicio → Variables
2. Añadir en formato `KEY=value`
3. Los cambios disparan redeploy automático

**Formato para variables anidadas en .NET:**
```bash
# Usar __ para separar niveles de anidación
JwtSettings__Secret=tu-secret
ResendSettings__ApiKey=re_...
ConnectionStrings__DefaultConnection=Host=...
AdminSeed__Email=admin@...
```

> Railway genera automáticamente `DATABASE_URL`. Si se usa PostgreSQL de Railway se puede referenciar como `${{DATABASE_URL}}` en `ConnectionStrings__DefaultConnection`.

```bash
# Connection string completo con SSL
```

## User Secrets (Desarrollo Local)

Alternativa más segura a `appsettings.Development.json` — los valores se guardan fuera del repositorio en `~/.microsoft/usersecrets/`.

```bash
cd backend/src/SimRacingShop.API

# Inicializar
dotnet user-secrets init

# Añadir secrets
dotnet user-secrets set "JwtSettings:Secret" "tu-secret-local"
dotnet user-secrets set "ResendSettings:ApiKey" "re_..."
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Host=localhost;..."
dotnet user-secrets set "AdminSeed:Email" "admin@simracingshop.com"
dotnet user-secrets set "AdminSeed:Password" "AdminPassword123!"

# Ver todos
dotnet user-secrets list

# Eliminar uno
dotnet user-secrets remove "JwtSettings:Secret"

# Limpiar todos
dotnet user-secrets clear
```

## Seguridad y Mejores Prácticas

### Hacer

1. **Usar User Secrets en desarrollo** — evita tener secrets en ficheros del repo
2. **Variables diferentes por entorno** — nunca reutilizar el `JwtSettings__Secret` entre dev y producción
3. **Stripe test en dev/staging, live solo en producción**
4. **Rotar secrets periódicamente** — JWT Secret cada 90 días en producción
5. **Verificar dominio en Resend** antes de enviar emails en producción

### No hacer

1. **NUNCA commitear secrets** — añadir a `.gitignore`:
   ```
   .env.local
   appsettings.Development.json
   appsettings.Production.json
   ```

2. **No exponer secrets en logs:**
   ```csharp
   // ❌
   logger.LogInformation($"Conectando con password: {password}");

   // ✅
   logger.LogInformation("Conectando a base de datos");
   ```

3. **No compartir secrets por canales inseguros** — usar 1Password / Bitwarden o regenerar tras compartir

## Checklist de Variables

### Antes del Primer Deploy

- [ ] `ConnectionStrings__DefaultConnection` configurado con SSL
- [ ] `ConnectionStrings__Redis` configurado
- [ ] `JwtSettings__Secret` robusto (32+ chars, diferente de dev)
- [ ] `AdminSeed__Email` y `AdminSeed__Password` configurados
- [ ] `ResendSettings__ApiKey` y dominio verificado en Resend
- [ ] `ResendSettings__FrontendBaseUrl` apunta a la URL de producción
- [ ] `Cors__AllowedOrigins` solo contiene el dominio real
- [ ] `Stripe__SecretKey` en live mode (`sk_live_...`)
- [ ] `NEXT_PUBLIC_API_URL` apunta a la URL pública del backend

### Tras el Primer Deploy

- [ ] Verificar que el admin seed se ha creado (`GET /api/auth/me` con sus credenciales)
- [ ] `AdminSeed__Password` puede eliminarse o cambiarse (el seeder ya no lo usará)
- [ ] Probar envío de email de verificación

## Troubleshooting

### "Cannot connect to database"
**Causa:** Connection string incorrecto o PostgreSQL no arrancado
**Solución:** Verificar `ConnectionStrings__DefaultConnection` y que Docker esté corriendo (`docker-compose up -d`)

### "JWT token invalid" / 401 inesperado
**Causa:** `JwtSettings__Secret` diferente entre quien emite y quien valida el token
**Solución:** Verificar que el mismo secret está en todos los entornos que deben comunicarse

### "No admin user exists and AdminSeed settings are not configured"
**Causa:** Primera ejecución sin configurar `AdminSeed`
**Solución:** Añadir `AdminSeed__Email` y `AdminSeed__Password` a las variables de entorno y reiniciar

### "CORS policy blocked"
**Causa:** La URL del frontend no está en `Cors.AllowedOrigins`
**Solución:** Añadir la URL exacta (con protocolo y puerto) al array de orígenes permitidos

### "Failed to send email"
**Causa:** `ResendSettings__ApiKey` inválida o `FromEmail` no verificado
**Solución:** Verificar la API key en el dashboard de Resend y confirmar que el dominio/email está verificado
